# 对标视频文稿改写

这个 skill 用来把中文对标短视频链接、原视频字幕、转写稿或参考文案，改写成适合自媒体发布的中文口播逐字稿。默认目标是 2-3 分钟，重点是保留强开头和爆点逻辑，同时替换例子、加入用户自己的观点，避免变成照搬原稿。

## 适合什么时候用

当你想按“对标视频文稿流程”处理一个参考视频或参考文案时使用：

- 获取原视频稿或参考文案结构
- 结构化转述原稿
- 保留原文开头的强痛点和吸引力
- 加入自己的观点、框架和例子
- 改写成中文短视频口播稿
- 生成标题、封面文案
- 完整结果默认归档到 `内容/`

## 默认流程

1. 先提取参考视频信息：标题、作者、标签、互动数据、正文、公开视频地址和字幕。
2. 如果是小红书链接，不能只因为可见页面打不开就放弃，必须至少尝试 3 种真实方法：
   - 正常打开页面，或通过搜索/缓存拿可见信息。
   - 联网抓取页面 HTML，检查内嵌数据。
   - 在页面数据里搜索 `noteDetailMap`、`mediaV2`、`subtitles`、`masterUrl`、`backupUrls`、`desc`、`title`、`nickname`、`tagList` 和互动数据。
   - 如果找到中文字幕或 source SRT 地址，下载 SRT，并在多个版本存在时互相核对。
   - 如果没有字幕但有视频地址，再走 ChatCut 转录或其他下载/转录路径。
   - 如果本机有 `yt-dlp`，可以用 `yt-dlp --dump-json` 探测视频元数据。
3. 拿到原稿后，不完整复制长文，而是按原顺序做结构化转述，只保留少量关键短句；即使用户说“直接改写”，也要输出原视频文案的结构化处理。
4. 如果用户提供自己的观点、例子或表达框架，优先使用用户版本。
5. 改写时保留参考开头的吸引力，替换案例和表达方式。
6. 输出标题与封面、逐字稿、参考链接和提取文案、用户原始想法。
7. 产出完整结果时，默认保存到当前工作区的 `内容/` 文件夹，除非用户明确说不要保存。

## 输出结构

归档文件默认使用：

```text
内容/YYYY-MM-DD_HHMM_帖子标题.md
```

文件结构：

```markdown
# 帖子标题

## 标题与封面

## 逐字稿

## 参考视频链接和提取出来的文案

## 我的原始想法
```

## 示例提示

```text
用 reference-video-script-rewriter 处理这个小红书对标视频。
先尽量提取原视频字幕，开头保持一致，例子不要完全跟原文一样，最后给我标题封面和逐字稿。
```

## 重要规则

- 小红书链接必须先尝试多种提取方式，至少 3 种真实方法失败后，才可以让用户粘贴字幕。
- 提取到字幕或转写稿时，必须输出原视频文案的结构化处理，不能只给改写稿。
- 完整成稿默认创建 md 归档，除非用户明确说不要保存。
- 不要全文复制第三方长文案，应该结构化转述并少量短摘。
- 用户说“直接改写”时，不要过度发散新主题，优先做文稿转述和节奏改写。
- 初始任务只做文稿，不默认进入录制、剪辑、字幕、导出视频流程。
